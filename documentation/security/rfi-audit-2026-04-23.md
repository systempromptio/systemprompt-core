### Documented Accepted Risk

**RUSTSEC-2023-0071 / GHSA-9c48-w39g-hm26 — Marvin Attack in `rsa` crate (LOW).**
Pulled transitively via `jsonwebtoken` for RSA-family JWT verification. No upstream fix available — the `rsa` crate does not yet offer constant-time implementations.
Mitigations in place:
1. Recommended signing algorithms for customer IdP tokens are ES256 / ES384 / EdDSA — these do not execute the vulnerable RSA path.
2. JWT verification is CPU-bounded and not an unauthenticated high-throughput endpoint, limiting exploitability of the timing side-channel.
3. Tracked — ignore is removed as soon as a fixed `rsa` release is available.

Full justification recorded inline in `deny.toml` under `[advisories].ignore`.

### Supply-chain Integrity on Release

- Release binaries are built in GitHub-hosted CI runners from tagged commits (`.github/workflows/release-sign.yml`).
- Signed with Sigstore `cosign` keyless, OIDC-bound to this repository and the release workflow.
- SHA256SUMS + per-file `.sig` + `.pem` bundle attached to every release.
- Verification command documented in `SECURITY.md`.

## 4b. Test Coverage and Quality Gates

The test and coverage investment is substantial and largely hidden behind the `crates/tests/` separate workspace. Summary for reviewers:

- **8,777 tests passing, 0 failures, 35 test crates.** Verified by `cargo test --manifest-path crates/tests/Cargo.toml --workspace` on `main`.
- **Eight test categories** organised under `crates/tests/`: `unit/`, `integration/`, `contract/`, `concurrency/`, `property/` (proptest), `fuzz/` (four targets: `a2a_request`, `config_loading`, `identifier_validation`, `jsonrpc_parse`), `loadtest/`, `bench/`.
- **Coverage tooling is operational.** A dedicated `crates/tests/.cargo/config.toml` overrides the root cranelift / sccache settings with the LLVM backend, which `cargo-llvm-cov` and `-Cinstrument-coverage` require. `just coverage`, `just coverage-html`, and `just coverage-clean` are the supported entry points.
- **Measured baseline (2026-04-02):** 25.97% line coverage across 30,067 production lines. The headline number includes large tracts of plain serde-typed `shared/models` (9,160 lines, 11.5% covered — dominated by struct and enum definitions where line coverage is a poor proxy for behavioural coverage).
- **Where coverage is highest** — and where it actually matters for a healthcare RFI:

| Crate | Coverage | Relevance |
|-------|----------|-----------|
| `infra/events` | **100.0%** | audit/event pipeline |
| `infra/security` | **96.8%** | JWT validation, auth extraction, manifest signing |
| `infra/config` | **86.7%** | secrets/profile bootstrap |
| `domain/templates` | 75.1% | |
| `shared/client` | 74.0% | |
| `infra/loader` | 71.5% | |

  Security-critical surfaces (`infra/events`, `infra/security`, `infra/config`) sit at 86–100%. The "headline" 25.97% is a denominator-of-everything number; the numerator where it counts is much higher.

- **CI integration** of the coverage tooling is the remaining gap. `.github/workflows/coverage.yml` publishes a job that runs `cargo-llvm-cov` on every push to `main` plus a weekly cron, producing `coverage-summary.{txt,json}` and `lcov.info` as artefacts and a per-run GitHub step summary. ## 5. Pre-answered Enterprise Security Questionnaire

Full pre-answers live in [compliance-control-matrix.md §4](compliance-control-matrix.md). Headline answers:

- **Certifications**: systemprompt.io as a vendor holds no SOC 2 / ISO 27001 / HITRUST certifications. The product is source-available infrastructure that the customer deploys inside their existing compliance boundary. Control-level support mappings provided.
- **BAA**: Not applicable — systemprompt as a vendor does not create, receive, maintain, or transmit PHI. Binary runs in the customer's environment.
- **Data location**: Customer's own Postgres, under customer control. The vendor never sees customer data.
- **Encryption**: TLS 1.2+ in transit (enforced at entry). Secrets-at-rest via the customer's envelope-encryption infrastructure (KMS / Vault / sops) — the binary receives plaintext only after the customer's tooling opens the envelope, so the master key never enters the binary. DB-level encryption at rest is customer-managed.
- **SSO**: OIDC through the customer's IdP.
- **SBOM**: CycloneDX, per release.
- **Release integrity**: cosign keyless, OIDC-bound to this repo, transparency-logged in Rekor.
- **Business continuity**: Source-available under BSL-1.1 with automatic conversion to Apache 2.0 after four years. Customer keeps indefinite usage rights under the licence.
